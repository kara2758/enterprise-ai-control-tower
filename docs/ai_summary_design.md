# AI Executive Summary — Design

This document describes the AI executive summary feature implemented in
`src/ai_summary.py` (Stage 7) and integrated into the Streamlit dashboard.

> ⚠️ The summary is generated from **synthetic data** for the fictional company
> **ABC Global Services**. It is for portfolio and demonstration purposes only.

---

## 1. Purpose

The AI Executive Summary turns the structured KPI results and risk register into
a concise, management-ready briefing: a short narrative, key risks, recommended
actions, management questions, and a confidence note. It is the bridge between
raw metrics and executive decision-making.

---

## 2. Demo Mode vs Live Mode

The module always works, with or without API access.

| | Demo Mode | Live Mode |
|---|-----------|-----------|
| **Trigger** | `force_demo=True`, or no `OPENAI_API_KEY` / `OPENAI_MODEL` | Both env vars present and `force_demo=False` |
| **AI call** | None — deterministic | Calls the OpenAI API |
| **Output** | Built from payload values | Generated by the model |
| **`mode` field** | `"demo"` | `"live"` (or `"fallback"` on error) |
| **API key needed** | No | Yes |

**Fallback:** if live mode is selected but the SDK is missing, credentials are
absent, the API call fails, or the response can't be parsed as JSON, the module
returns a deterministic demo summary with `mode = "fallback"` and a clear
`confidence_note`. It never crashes the dashboard.

Configuration is read from the environment via `python-dotenv` (`.env`). See
`.env.example` for the placeholders:

```
OPENAI_API_KEY=your_api_key_here
OPENAI_MODEL=your_model_here
```

---

## 3. Why Only Aggregated Data Is Sent to AI

`build_executive_summary_payload` constructs a **compact** payload containing
only:

- `executive_kpis`
- `risk_summary`
- the **top 10** risk records
- invoice / ticket / approval KPI highlights

The full CSV datasets are **never** sent to the API. This minimizes data
exposure, reduces token usage and cost, and keeps the request focused on what
matters for an executive briefing.

---

## 4. Output Structure

All modes return the same dictionary shape:

```json
{
  "mode": "demo | live | fallback",
  "executive_summary": "short narrative paragraph",
  "key_risks": ["...", "..."],
  "recommended_actions": ["...", "..."],
  "management_questions": ["...", "..."],
  "confidence_note": "..."
}
```

`format_summary_for_display(summary)` renders this as clean markdown for
Streamlit.

---

## 5. Dashboard Integration

The dashboard adds an **AI Executive Summary** section between the Executive
Overview and the charts:

- A short explanation of the feature.
- A **"Use demo mode"** checkbox (default: True).
- A **"Generate Executive Summary"** button.

The summary is generated **only when the button is clicked** — never
automatically on page load. The mode (demo / live / fallback) is shown clearly,
followed by the formatted summary.

---

## 6. Safety & Privacy Notes

- API keys are read from the environment only; they are **never hardcoded,
  printed, logged, or returned**.
- `.env` is git-ignored; only `.env.example` (placeholders) is committed.
- Only aggregated KPIs and top-risk records are sent to the API.
- No external API is called at import time, and live mode runs only on explicit
  invocation (the dashboard button or a direct function call).
- Error handling avoids leaking exception details that could contain sensitive
  information — fallbacks reference only the error type.
