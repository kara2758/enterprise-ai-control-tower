# Dataset Dictionary — Enterprise AI Control Tower

This document describes the synthetic datasets used by the Enterprise AI Control
Tower for the fictional company **ABC Global Services**.

> ⚠️ **All data is synthetic.** It is generated by
> [`scripts/generate_synthetic_data.py`](../scripts/generate_synthetic_data.py)
> and contains no real company, customer, vendor, invoice, or personal data.

**Fixed reference date:** `2026-06-22`
All "as of today" fields (e.g. `days_overdue`, `waiting_days`, SLA status) are
calculated relative to this fixed date so the datasets are fully reproducible.

**Reproducibility:** the generator uses a fixed random seed (`42`). Running it
again reproduces identical datasets.

---

## 1. `invoices.csv`

**Purpose:** Accounts-payable invoices used for spend visibility, overdue
tracking, and approval-bottleneck monitoring.

| Column | Type | Description |
|--------|------|-------------|
| `invoice_id` | string | Unique invoice identifier (e.g. `INV-10001`). |
| `vendor_name` | string | Fictional vendor / supplier name. |
| `department` | string | Owning department (Finance, Operations, Procurement, IT, Customer Support, HR). |
| `invoice_date` | date | Date the invoice was issued. |
| `due_date` | date | Payment due date (`invoice_date` + net terms). |
| `amount_usd` | float | Invoice amount in USD equivalent. |
| `currency` | string | Original currency (USD, EUR, GBP). |
| `approval_status` | string | Pending, Approved, Rejected, Under Review. |
| `payment_status` | string | Paid, Unpaid, Partially Paid, On Hold. |
| `assigned_approver` | string | Fictional approver name. |
| `days_overdue` | int | Days past `due_date` as of the reference date (0 if paid or not yet due). |
| `invoice_category` | string | Spend category (Software, Consulting, Logistics, etc.). |
| `priority_level` | string | Low, Medium, High, Critical. |

**Example business use:** Flag invoices overdue by more than 7 days, or
high-value invoices (> 50,000 USD) still awaiting approval.

---

## 2. `customer_tickets.csv`

**Purpose:** Customer support tickets used for SLA monitoring, service-quality
tracking, and escalation management.

| Column | Type | Description |
|--------|------|-------------|
| `ticket_id` | string | Unique ticket identifier (e.g. `TCK-20001`). |
| `customer_name` | string | Fictional customer name. |
| `department` | string | Department associated with the ticket. |
| `issue_type` | string | Billing Issue, Technical Problem, Service Delay, Account Request, Complaint, Data Correction. |
| `priority` | string | Low, Medium, High, Critical. |
| `opened_date` | datetime | When the ticket was opened. |
| `sla_deadline` | datetime | SLA resolution deadline (`opened_date` + SLA hours by priority). |
| `resolved_date` | datetime | When resolved (empty if still open). |
| `status` | string | Open, In Progress, Resolved, Escalated, Waiting for Customer. |
| `assigned_team` | string | Team handling the ticket. |
| `response_time_hours` | float | Hours until first response. |
| `resolution_time_hours` | float | Hours until resolution (empty if unresolved). |
| `sla_status` | string | Within SLA, At Risk, Breached. |
| `customer_segment` | string | Enterprise, SMB, Startup, Government, Individual. |

**SLA targets by priority:** Critical = 8h, High = 24h, Medium = 48h, Low = 72h.

**Example business use:** Surface tickets that have breached SLA, or open tickets
"At Risk" (within ~12 hours of their deadline) so they can be prioritized.

---

## 3. `approval_requests.csv`

**Purpose:** Approval-workflow requests used to detect process bottlenecks and
delayed high-impact decisions.

| Column | Type | Description |
|--------|------|-------------|
| `request_id` | string | Unique request identifier (e.g. `REQ-30001`). |
| `request_type` | string | Purchase Approval, Payment Approval, Contract Review, Budget Request, Vendor Onboarding, Refund Approval. |
| `department` | string | Requesting department. |
| `requester` | string | Fictional requester name. |
| `approver` | string | Fictional approver name. |
| `requested_date` | date | When the request was submitted. |
| `approval_deadline` | date | Target decision date. |
| `approval_status` | string | Pending, Approved, Rejected, Escalated. |
| `waiting_days` | int | Days the request has been (or was) waiting for a decision. |
| `amount_usd` | float | Monetary value of the request, if applicable. |
| `business_impact` | string | Low, Medium, High, Critical. |
| `bottleneck_flag` | bool | True when a Pending/Escalated request has waited more than 5 days. |

**Example business use:** Identify approval bottlenecks (waiting > 5 days) or
critical-impact requests still pending a decision.

---

## 4. `risk_rules.csv`

**Purpose:** A simple, human-readable rules reference table that defines the risk
conditions the platform will evaluate in later stages. (No rule *logic* is
implemented yet — this stage only defines the rules as data.)

| Column | Type | Description |
|--------|------|-------------|
| `rule_id` | string | Unique rule identifier (e.g. `R-001`). |
| `rule_name` | string | Short human-readable rule name. |
| `data_source` | string | The dataset the rule applies to. |
| `condition_description` | string | Plain-language / pseudo-condition the rule checks. |
| `risk_level` | string | Low, Medium, High, Critical. |
| `recommended_action` | string | Suggested response when the rule triggers. |

**Included rules:**

| Rule | Source | Risk Level |
|------|--------|-----------|
| Overdue Invoice (> 7 days) | invoices.csv | High |
| High-Value Pending Invoice (> 50,000 USD) | invoices.csv | Critical |
| Ticket SLA Breached | customer_tickets.csv | High |
| Ticket Close to SLA Deadline | customer_tickets.csv | Medium |
| Approval Bottleneck (waiting > 5 days) | approval_requests.csv | Medium |
| Critical Request Still Pending | approval_requests.csv | Critical |

---

## Notes on Synthetic Data

- All names (vendors, approvers, requesters, customers) are **fictional** and
  randomly assembled — they do not represent real organizations or individuals.
- Values are randomly generated within plausible business ranges; they are **not**
  derived from any real source.
- The data intentionally contains **anomalies** (overdue invoices, breached
  SLAs, approval bottlenecks) so that later KPI and risk modules have meaningful
  signals to detect.
- To regenerate the datasets, run:
  ```bash
  python scripts/generate_synthetic_data.py
  ```
